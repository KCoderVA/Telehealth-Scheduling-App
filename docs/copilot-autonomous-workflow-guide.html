Workflow 1: User-Reported Bug/Feature/Issue Lifecycle

üìã Overview

This workflow covers the complete lifecycle of handling user-reported bugs, feature requests, or issues from initial intake through resolution and publication. Emphasis is on using Copilot Agent mode and @githubpr participant for maximum automation.

Workflow Stages:

  • Intake (Steps 1-3): Receive, document, and create GitHub issue
  • Investigation (Steps 4-6): Analyze root cause, plan fix, implement changes
  • Resolution (Steps 7-9): Document solution, close issue, publish updates

üéØ Stage 1: Issue Intake & Triage

Step 1: Receive Bug/Feature Report from User
Users report issues through various channels (email, Teams, in-person). Standardizing intake ensures no details are lost and creates audit trail for VA compliance.
Gathering structured information about the issue including: reporter contact info, severity, description, reproduction steps, screenshots, and expected vs actual behavior.
Information Collection:
  1. When user reports issue via email/Teams/in-person, note these details:
    • Reporter Info: Name, email, department, phone
    • Severity: Critical / High / Medium / Low
    • Issue Type: Bug / Feature Request / Enhancement
    • Title: Brief summary (under 80 characters)
    • Description: Detailed explanation of the issue
    • Steps to Reproduce: Numbered steps to recreate the problem
    • Screenshots: Visual evidence (if applicable)
    • Expected Behavior: What should happen
    • Actual Behavior: What actually happens
  2. If user provides incomplete info, ask clarifying questions immediately
  3. Save screenshots to s:\path\to\screenshots\ or project's assets\images\ folder
Telehealth Project Specific: For room booking issues, also capture: building name, room number, date/time of booking, manager involved in approval chain.
Step 2: Use VS Code Task to Create GitHub Issue
Automates GitHub issue creation with proper formatting, labels, and metadata without leaving VS Code. Eliminates manual browser-based issue creation.
Executes workspace-configured PowerShell script that uses GitHub API to create properly formatted issue with intake form data.
Using VS Code Task (Preferred Method):
  1. Press Ctrl+Shift+P ‚Üí Type Tasks: Run Task
  2. Select: üÜï Intake: Create Issue
  3. Task will prompt for each field:
    • GitHub Token (or press Enter to use browser-based auth)
    • Employee Name
    • Employee Email
    • Department
    • Phone Number
    • Severity (Critical/High/Medium/Low)
    • Issue Title
    • Issue Description
    • Reproduction Steps
    • Additional Notes
    • Screenshot Path (optional)
  4. Task creates issue and outputs: Issue number, URL, confirmation
  5. Copy issue number for next steps (e.g., #42)
Alternative: Use Copilot Chat: 
@githubpr create new issue with title "Room booking fails for Aurora 120" and labels "bug,intake"
No Task Configured? Use @githubpr directly in Copilot Chat: 
@githubpr create issue
Title: [paste title]
Labels: bug, intake
Body: [paste full description with reporter info]
Pro Tip: Set environment variable $env:GITHUB_TOKEN in your PowerShell profile to avoid token prompts. Generate token at github.com/settings/tokens with repo scope.
Step 3: Initial Triage with Copilot Agent
Copilot can analyze the issue description and workspace context to suggest likely root causes, affected files, and investigation starting points before you manually search.
Uses Copilot Agent mode with @workspace context to perform intelligent triage, identify related code areas, and create investigation plan.
Triage Process:
  1. Open Copilot Chat (Ctrl+Alt+I)
  2. Set mode to Agent
  3. Use this prompt pattern: 
    @workspace I received issue #42 about "Room booking fails for Aurora 120"

User reports: [paste key details from issue]

Please:
1. Search the codebase for related files
2. Identify likely root cause areas
3. List files I should examine
4. Suggest investigation approach
  4. Agent will autonomously:
    • Search workspace for relevant keywords
    • Identify files containing room booking logic
    • Analyze error patterns in code
    • Provide structured investigation plan
  5. Review Agent's findings and proceed to investigation stage
For Power Platform Projects: Include context about whether issue is in Canvas App, Power Automate flow, or SharePoint: 
@workspace Issue #42 is related to Canvas App room selection dropdown. Search Power Fx formulas in src/power-apps/ for room filtering logic.
After triage, add a comment to the GitHub issue with Copilot's findings: @githubpr comment on issue #42 "Initial triage suggests issue in room filtering logic (files: src/power-apps/Screens/RoomSelection.fx)"

üîç Stage 2: Investigation & Implementation

Step 4: Deep Dive Investigation with Copilot Agent
Copilot Agent can autonomously read multiple files, trace logic flows, identify bugs, and understand complex code relationships faster than manual investigation.
Uses Agent mode to perform comprehensive code analysis across the workspace, following function calls, examining data flows, and pinpointing the exact location and cause of issues.
Investigation Process:
  1. In Copilot Chat (Agent mode), use investigation prompt: 
    @workspace Investigate issue #42: Room booking fails for Aurora 120

Based on triage findings, examine:
1. Room filtering logic in src/power-apps/Screens/RoomSelection.fx
2. Data source queries for room availability
3. Any hardcoded building/room references
4. Validation rules that might reject "Aurora 120"

Please analyze the code and identify the root cause.
  2. Agent will autonomously:
    • Read relevant source files
    • Search for patterns and references
    • Trace logic flows across multiple files
    • Identify the specific line(s) causing the issue
    • Explain the root cause in detail
  3. Review Agent's analysis and ask follow-up questions if needed: 
    Why would the filter reject "Aurora 120" specifically? Is it a case sensitivity issue?
For Multi-Component Issues: Ask Agent to check interactions between components: 
@workspace Check if this is a Canvas App issue, Power Automate flow issue, or SharePoint list schema issue. Trace the data flow from user input through all systems.
Step 5: Implement Fix with @workspace Context
Copilot Agent can implement fixes across multiple files while maintaining code style, following project patterns, and ensuring consistency with existing codebase.
Uses Agent mode to autonomously write code changes, update configurations, and modify files to resolve the identified issue while preserving existing functionality.
Fix Implementation:
  1. Request fix implementation from Agent: 
    @workspace Fix issue #42

Please:
1. Update room filtering logic to be case-insensitive
2. Add defensive checks for null/undefined values
3. Include inline comments explaining the fix
4. Follow existing Power Fx code style in this workspace
  2. Agent will autonomously:
    • Locate the exact code requiring changes
    • Generate fix using proper syntax and patterns
    • Apply changes to appropriate file(s)
    • Add explanatory comments
    • Show you diff preview for review
  3. Review proposed changes in diff editor
  4. Accept or request modifications: 
    The fix looks good but can you also add error logging for debugging future issues?
  5. Agent will iterate on the fix until you approve
Multi-file Fixes: Agent can handle complex changes across multiple files. Just describe what needs to change: 
@workspace Update both the Canvas App filtering logic AND the SharePoint list schema to ensure room names are stored consistently in Title Case format.
Power Platform Specific Fixes:
  • Canvas App Formula Fix: Agent edits unpacked .fx files in src/power-apps/
  • Power Automate Flow Fix: Agent modifies flow definition JSON in src/power-automate/
  • SharePoint Schema Fix: Agent updates list schema XML in src/sharepoint/
Note: For web-based editor changes (Canvas Apps/Flows), you'll need to manually apply Agent's suggestions in PowerApps Studio or Power Automate designer, then export updated artifacts.

Testing verifies the fix resolves the reported issue without introducing regressions or breaking existing functionality.

  3. Document test results:

For VA healthcare projects, always test in a non-production environment first. Never apply fixes directly to production Power Platform environments without proper testing and approval.

WHY: Binary .msapp files cannot be reviewed, diffed, or merged in Git. Unpacking converts them to human-readable source files (.fx formulas, .json metadata, .yaml configs) for proper version control.
WHAT: Uses pac CLI to decompose .msapp binary into directory structure of source files that can be reviewed in VS Code and tracked in Git.
HOW:
  1. Download/export .msapp file from PowerApps Studio to src/power-apps/
  2. In VS Code Copilot Chat, type: @terminal unpack TelehealthBookingApp.msapp to source files
  3. Copilot generates and executes: 
    pac canvas unpack --msapp ./src/power-apps/TelehealthBookingApp.msapp --sources ./src/power-apps/TelehealthBookingApp/Source
  4. Verify unpacked structure appears:
    • Src/ - Screen and control definitions (.fx Power Fx formulas)
    • DataSources/ - Connection metadata (.json)
    • Properties/ - App properties and settings (.json)
    • Resources/ - Images, icons, media files
  5. Delete original .msapp file (not needed for version control)
  6. Commit unpacked source files to Git
BEST PRACTICE: Add .msapp files to .gitignore to prevent accidentally committing binary files. Only version control unpacked source directories.

Step 3: Pack Source Files to .msapp

WHY: To deploy or import Canvas Apps, you need the .msapp binary format. Packing rebuilds the .msapp from version-controlled source files for deployment to other environments.
WHAT: Uses pac CLI to compile source directory structure back into a deployable .msapp package for import into Power Platform environments.
HOW:
  1. In VS Code Copilot Chat, type: @terminal pack source files to deployable .msapp
  2. Copilot generates and executes: 
    pac canvas pack --sources ./src/power-apps/TelehealthBookingApp/Source --msapp ./releases/TelehealthBookingApp_v0.9.5.msapp
  3. Verify .msapp created in releases/ directory
  4. Test .msapp by importing to test environment via PowerApps Studio
  5. Store release .msapp in releases/v0.9.x/ with version number in filename
TIP: Use semantic versioning in .msapp filenames (e.g., TelehealthBookingApp_v0.9.5.msapp) to track releases alongside Git tags and CHANGELOG entries.

Step 4: Solution Export & Import Automation

WHY: Solutions package multiple Power Platform components (apps, flows, entities) together for coordinated deployment across environments (Dev ‚Üí Test ‚Üí Prod).
WHAT: Uses pac CLI to export complete solution packages from source environment and import them into target environments with proper dependency resolution.
HOW:

    • In VS Code Copilot Chat: @terminal export TelehealthBooking solution as managed
    • Copilot executes:

  2. Import Solution to Target Environment:

    • Validate deployment in target environment via Power Platform admin center

This section provides solutions to common problems encountered during autonomous Copilot workflows. Use these troubleshooting patterns to quickly resolve issues and maintain development momentum.

Issue 1: GitHub Authentication Failures

SYMPTOMS:
  • @githubpr participant returns "Not authenticated" or "Unable to access repository" errors
  • VS Code shows "Sign in to use GitHub Pull Requests" notification
  • GitHub tasks fail with 401 Unauthorized errors
  • Cannot create issues, read PRs, or access repository information
SOLUTION:
  1. Re-authenticate GitHub Account:
    • Click bottom-left account icon in VS Code ‚Üí "Sign in to Sync Settings"
    • Select "Sign in with GitHub"
    • Browser opens for GitHub authorization
    • Grant VS Code permissions to access repositories
    • Return to VS Code, verify account name appears in status bar
  2. Verify GitHub Extension Status:
    • Open Extensions view (Ctrl+Shift+X)
    • Search for "GitHub Pull Requests and Issues"
    • Verify extension is enabled (not disabled or requiring reload)
    • Check extension version (update if outdated)
  3. Clear GitHub Credentials Cache:
    • Press Ctrl+Shift+P ‚Üí "GitHub: Sign Out"
    • Press Ctrl+Shift+P ‚Üí "GitHub: Sign In"
    • Complete authentication flow again
  4. Verify Repository Permissions:
    • Visit https://github.com/settings/tokens in browser
    • Check personal access token has required scopes: repo, read:org
    • If token expired, generate new token and re-authenticate VS Code
PREVENTION TIP: Enable "Trust this extension" for GitHub Pull Requests extension in workspace settings to avoid re-authentication prompts.

Issue 2: Copilot Extension Conflicts

SYMPTOMS:
  • Copilot Chat participant (@workspace, @githubpr) not available in suggestions
  • "Copilot is not responding" or timeout errors
  • Chat panel opens but shows "Unable to load participants" message
  • Extension icon shows warning or error indicator
SOLUTION:
  1. Reload VS Code Window:
    • Press Ctrl+Shift+P ‚Üí "Developer: Reload Window"
    • Wait for all extensions to reload
    • Test Copilot Chat (Ctrl+Alt+I) with simple query
  2. Check Extension Compatibility:
    • Open Extensions view (Ctrl+Shift+X)
    • Search for "GitHub Copilot Chat"
    • Check "Details" tab for known compatibility issues
    • Verify VS Code version meets minimum requirements (usually latest stable)
    • Update VS Code if needed: Help ‚Üí Check for Updates
  3. Disable Conflicting Extensions:
    • Common conflicts: other AI assistants (Tabnine, Kite), alternative chat extensions
    • Temporarily disable suspected conflicting extensions
    • Reload window and test Copilot functionality
    • Re-enable extensions one at a time to identify conflict
  4. Clear Extension Cache:
    • Press Ctrl+Shift+P ‚Üí "Developer: Reload Window with Extensions Disabled"
    • Press Ctrl+Shift+P ‚Üí "Developer: Reload Window"
    • This forces extensions to reinitialize from clean state
WARNING: If issues persist after these steps, uninstall and reinstall GitHub Copilot Chat extension. Export your workspace settings first to preserve configurations.

Issue 3: Workspace Context Not Available

SYMPTOMS:
  • @workspace participant returns "No workspace context found" or generic responses
  • Copilot doesn't recognize files, functions, or project-specific code
  • Semantic search returns empty or irrelevant results
  • Code suggestions don't match project patterns or conventions
SOLUTION:
  1. Verify Workspace is Open:
    • Check bottom-left corner shows workspace name (not just folder name)
    • If folder is open instead of workspace: File ‚Üí Open Workspace from File
    • Select Telehealth Resources Project.code-workspace
    • VS Code reloads with full workspace context
  2. Rebuild Workspace Index:
    • Press Ctrl+Shift+P ‚Üí "Developer: Reload Window"
    • Wait 30-60 seconds for indexing to complete (check bottom status bar for progress)
    • Test @workspace with specific query: "Find the room booking form in Canvas App"
  3. Check .gitignore and File Exclusions:
    • Open .gitignore file in workspace root
    • Verify important source files are NOT excluded (e.g., src/ directories)
    • Check VS Code settings: Files ‚Üí Exclude patterns
    • Remove exclusions that hide relevant code from Copilot indexing
  4. Provide Explicit Context in Queries:
    • Instead of: "How does booking work?"
    • Use: "@workspace How does the room booking workflow work in TelehealthBookingApp Canvas App?"
    • Reference specific files: "@workspace Explain the formula in src/power-apps/.../galRoomAvailability.Items"
BEST PRACTICE: Keep workspace file structure organized with clear directory naming. Copilot indexes based on file names, paths, and content - descriptive names improve context quality.

Issue 4: pac CLI Authentication Errors

SYMPTOMS:
  • pac canvas unpack or pac solution export fails with "Not authenticated" error
  • pac auth list shows no authentication profiles
  • Commands return "Unable to connect to Power Platform environment" messages
  • Authentication browser window opens but doesn't complete login flow
SOLUTION:
  1. Create New Authentication Profile:
    • In VS Code terminal, run: 
      pac auth create --environment https://[your-environment].crm.dynamics.com
    • Browser opens for interactive login
    • Use correct M365 credentials (same as PowerApps Studio access)
    • Grant permissions when prompted
    • Verify success: pac auth list shows new profile with green checkmark
  2. Select Correct Authentication Profile:
    • If multiple profiles exist: pac auth select --index [number]
    • Or use named profile: pac auth select --name "Dev Environment"
    • Verify active profile: pac org who (displays current environment)
  3. Clear Expired Tokens:
    • Delete expired profile: pac auth delete --index [number]
    • Recreate with fresh authentication: pac auth create
    • Tokens typically expire after 90 days - re-authenticate periodically
  4. Verify Environment URL:
    • Find correct URL: Open https://make.powerapps.com
    • Click gear icon ‚Üí Session details ‚Üí Instance url
    • Use exact URL format: https://[org-name].[region].dynamics.com
    • For VA Gov Cloud: https://[org-name].crm9.dynamics.com
TIP: Save environment URLs in VS Code workspace settings JSON for quick reference: 
{
    "powerPlatform.environments": {
        "dev": "https://dev-hines.crm9.dynamics.com",
        "test": "https://test-hines.crm9.dynamics.com",
        "prod": "https://hines.crm9.dynamics.com"
    }
}

Issue 5: Canvas App Unpacking Failures

SYMPTOMS:
  • pac canvas unpack returns error: "The .msapp file is corrupted or invalid"
  • Unpacking completes but source files are missing or incomplete
  • Error: "Unable to parse .msapp file format" or version compatibility issues
  • Unpacked files contain unreadable binary or garbled content
SOLUTION:
  1. Verify .msapp File Integrity:
    • Check file size (should be several MB for typical Canvas App)
    • Re-export from PowerApps Studio: File ‚Üí Export ‚Üí Package (.msapp)
    • Use "Save As" download (not browser preview) to avoid truncation
    • Verify download completed successfully (no partial downloads)
  2. Update pac CLI to Latest Version:
    • Check current version: pac -v
    • Update with .NET tool: dotnet tool update -g Microsoft.PowerApps.CLI.Tool
    • Or download installer from https://aka.ms/PowerAppsCLI
    • Verify updated: pac -v (should show latest version, e.g., 1.30+)
    • Retry unpacking with updated CLI
  3. Use Clean Output Directory:
    • Delete existing unpacked source directory if present
    • Create fresh output directory: mkdir ./src/power-apps/TelehealthBookingApp/Source
    • Run unpack command with clean destination: 
      pac canvas unpack --msapp [input.msapp] --sources [clean-directory]
  4. Check Canvas App Format Version:
    • In PowerApps Studio, click File ‚Üí Settings ‚Üí Upcoming features
    • Disable experimental features that may cause compatibility issues
    • Save and re-export .msapp
    • Older pac CLI versions may not support newest app format versions
WARNING: Always keep a backup copy of working .msapp files before unpacking. Unpacking process is non-destructive, but having backup ensures you can restore if issues occur.

Issue 6: Git Merge Conflicts in Power Fx Files

SYMPTOMS:
  • Git shows merge conflicts in .fx files (Power Fx formulas) after pulling changes
  • VS Code highlights conflict markers in Canvas App source files
  • Unable to pack .msapp due to invalid .fx syntax from unresolved conflicts
  • Multiple developers edited same control properties/formulas simultaneously
SOLUTION:
  1. Identify Conflicting Files:
    • Run: git status to see conflicted files
    • Look for files in src/power-apps/.../Src/ directory (control definitions)
    • Open conflicted .fx files in VS Code (conflicts shown with <<<<<<< markers)
  2. Analyze Conflict Context:
    • Use @workspace in Copilot Chat: "Explain this Power Fx merge conflict and recommend resolution"
    • Copilot analyzes both formula versions and suggests which to keep or how to merge
    • Consider business logic: which version implements correct behavior?
  3. Resolve with VS Code Merge Editor:
    • Click "Resolve in Merge Editor" button in VS Code
    • Review "Current" (your changes) vs "Incoming" (pulled changes)
    • Select appropriate version or manually combine formulas
    • Validate Power Fx syntax (no unmatched parentheses, correct function names)
    • Click "Complete Merge" button
  4. Test Merged Canvas App:
    • Pack source files to .msapp: pac canvas pack --sources [src-dir] --msapp [output.msapp]
    • Import to test environment via PowerApps Studio
    • Test affected controls/screens to ensure formulas work correctly
    • If errors, review and adjust merged formulas
  5. Commit Resolved Conflict:
    • Stage resolved files: git add [conflicted-files]
    • Complete merge: git commit -m "Resolved merge conflict in [control-name] formula"
    • Push to remote: git push
BEST PRACTICE: To minimize merge conflicts, coordinate Canvas App editing among team members. Use feature branches for separate screens/components, and communicate via GitHub issues before editing shared controls.

Issue 7: Copilot Response Length Limits

SYMPTOMS:
  • Copilot responses are truncated mid-sentence or mid-code block
  • Large code generation requests return incomplete implementations
  • Documentation requests cut off without all sections completed
  • Error message: "Response too long, please try a more specific query"
SOLUTION:
  1. Break Large Requests into Smaller Tasks:
    • Instead of: "Create complete Canvas App with all screens and formulas"
    • Use: "Create home screen layout" ‚Üí "Add room booking form" ‚Üí "Implement conflict detection formula"
    • Complete one section, verify, then request next section
  2. Ask for Continuation:
    • When response truncates, immediately type: "Continue from where you stopped"
    • Copilot resumes from truncation point with remaining content
    • Repeat until complete content received
  3. Request Outline First, Then Details:
    • First query: "Provide outline of steps for implementing room booking workflow"
    • Review outline, select specific step
    • Follow-up query: "Implement step 3 in detail with complete code"
    • Iterate through each step individually
  4. Use File References Instead of Inline Content:
    • For large code: "Create the formula in src/power-apps/formulas/roomFiltering.fx"
    • Copilot creates separate file instead of inline response (avoids length limits)
    • Review file content in VS Code editor (no response length restrictions)
TIP: For comprehensive documentation like this guide, explicitly request: "Break this into multiple responses/sections to avoid length limits". Copilot will plan incremental delivery.

Issue 8: Power Automate Flow Import Errors

SYMPTOMS:
  • Imported flow shows errors for connections or missing dependencies
  • Actions marked with warning: "Connection not configured"
  • Flow won't save: "Invalid action configuration" or "Missing required parameters"
  • Environment-specific references fail after importing to different environment
SOLUTION:
  1. Reconfigure Connections After Import:
    • Open imported flow in edit mode
    • Each connection action shows warning icon
    • Click action ‚Üí Select connection dropdown ‚Üí "Add new connection"
    • Authenticate with target environment credentials
    • Repeat for all connection actions (SharePoint, Office 365, Approvals, etc.)
  2. Update Environment-Specific References:
    • SharePoint site URLs: Update to target environment's SharePoint
    • List/library names: Verify they exist in target environment with same schema
    • User emails/groups: Update to target environment's users (test vs prod accounts)
    • Canvas App IDs: If flow launches app, update app reference to target environment's version
  3. Use Connection References in Flows:
    • When creating flows, use solution-aware flows with connection references
    • Connection references abstract environment-specific details
    • During solution import, configure connection references once (not in every action)
    • Future imports automatically use configured connection references
  4. Validate Flow Before Activation:
    • After reconfiguring, click "Flow checker" (top-right)
    • Resolve all errors and warnings
    • Test flow with sample data before turning on
    • Monitor first live execution to confirm correct behavior
BEST PRACTICE: Package flows in solutions with connection references and environment variables. This reduces configuration effort when deploying across Dev/Test/Prod environments.
